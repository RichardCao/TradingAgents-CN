# 项目谱系说明

本文档用于说明本仓库与上游项目的关系，避免在持续演进过程中埋没上游贡献，也避免把本仓库的独立增强误写成“仅做语言替换”。

## 1. 当前关系

- 上游项目：`TauricResearch/TradingAgents`
- 中文分支项目：`hsliuping/TradingAgents-CN`
- 当前维护仓库：基于 `TradingAgents-CN` 的 fork 持续维护

本仓库并不是一个“从零独立重写”的新项目，也不只是“把原仓库文本翻译成中文”的轻量镜像，而是在中文 fork 基础上持续加入：

- 中文配置与中文报告适配
- 本地部署、数据源与运行环境修复
- MongoDB / Redis / 前后端联调改造
- 自选股、同步历史、数据清理、标签管理等本地增强

## 2. 代码来源的记录原则

后续代码变更按以下三类理解：

- 上游回补（backfill）
  - 参考上游的新能力、修复思路或接口设计，在本仓库内按现有架构局部落地。
  - 目标是吸收有价值的能力，而不是机械同步所有上游改动。
- 本地增强（feat）
  - 围绕中文仓库的真实使用场景新增功能、交互或运维能力。
- 本地修复（fix）
  - 修复本仓库已有问题，提升稳定性、可观测性和可维护性。

## 3. 为什么不直接全量追上游

当前仓库已经存在大量与上游不同的现实约束，例如：

- 中文前后端页面与交互
- 本地数据库与缓存集成
- 多数据源适配
- 中文报告输出与导出链路

因此更合理的策略是：

- 持续跟踪上游
- 只回补高价值、低冲突、可验证的能力
- 不为了“看起来同步”而破坏当前可运行状态

## 4. 对外描述建议

对外说明本仓库时，建议使用以下口径：

- “基于 `TradingAgents-CN` fork 持续维护”
- “部分能力参考上游 `TradingAgents` 的后续演进并做局部回补”
- “包含中文化、部署修复、数据源适配与本地工程化增强”

不建议使用以下容易引发误解的说法：

- “完全原创，与上游无关”
- “只是原项目换成中文”

## 5. 提交与公开发布边界

公开提交时应继续遵守：

- 保留原项目许可证、版权声明与项目谱系信息
- 不提交本地 `.env`、密钥、数据库配置和机器路径
- 在 changelog、release note 和提交说明中，尽量区分 `backfill / feat / fix / docs`

这样既能尊重上游贡献，也能清晰体现本仓库持续维护与独立增强的工作量。
